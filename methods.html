§0 全体ビジョン

「構造・道具・評価・知識を与え、手続きは与えない」 — AI に研究をさせるために、固定的なパイプラインではなく、エージェントが自律的に判断して動き続ける環境を設計する。

取り組みの目標

  • 既存 AI（Transformer ベース）を超える革新的技術を AI 自身に発見させる
  • 研究自動化そのものの改善を AI に研究させる（メタ的ループ）
  • 上記の検証基盤として、汎用 ML 研究能力を持つエージェントを構築する

4 コンポーネントの位置づけ

「論文を読む → 再現する → 改善する → 評価する」という研究のサイクルを 4 つの独立したツールで分解。それぞれが個別に動き、組み合わせると end-to-end になる設計。

PaperBench Iter「生成したコードを評価し、失敗から学んで再生成する」反復改善ループの実証
Self-Improving Agent「弱いベースラインから自動で改善し続ける」24/365 自己改善ループの最小実装
Reproduce PipelinearXiv URL から再現結果までを 5 ステージで自動化する end-to-end パイプライン
Literature Scout研究コンテキストに対する関連論文の自動発見ツール（pull 型）

§1 PaperBench Iteration

タスク
論文（Stochastic Interpolants）の rubric（評価項目ツリー）に対して、生成したコードをどれだけ満たせるかを自動評価し、不足を分析して改善コードを再生成する。これを反復することで PaperBench スコアを伸ばす。

提案手法 — generate-then-iterate

多くの再現エージェントは「1 回コードを生成 → 評価」で止まる。我々は「生成 → 評価 → 失敗分析 → 改善コード生成」を反復することでスコアを引き上げる。

各反復のステップ

  1. Stage 2（Code Finding）で論文に対する初期コードを生成（v0 baseline）
  2. PaperBench 公式の SimpleJudge（claude-sonnet-4 ベース）が rubric の 58 ノードを採点
  3. 低スコアのノードを抽出し、その reasoning を読んで「何が足りなかったか」を特定
  4. 論文と既存コードを Claude に再投入して改善コードを生成（v1, v2）
  5. 再評価 → スコア推移を記録

各反復で具体的に変えたこと

反復主な変更点
v0 → v1ImageNet データセットアクセス（HuggingFace）/ 64-tile マスク戦略 / 結合公式の修正 / step ベース訓練（200,000 steps）/ クラス条件付け
v1 → v2Dopri5 ODE ソルバの復元（torchdiffeq）/ FID 評価コード追加 / 超解像のダウンサンプリング処理 / Uncoupled モデルの分離 / ODE 関数のラッピング

評価方法

PaperBench 公式の hierarchical scoring を使用。58 ノードのツリーで、各ノードに重みが付き、子ノードのスコアが親に伝播する。0-25 / 26-50 / 51-75 / 76-100 の 4 段階で集計。

結果が示すこと

  • 1 回生成（v0）の 43.2% は IterativeAgent o1-high 平均 43.4% とほぼ同等。初期生成は SOTA レベル
  • 2 回反復（v2）の 94.5% は同論文において SOTA を超える結果（ただし最易論文1本での結果なので公平な比較ではない）
  • 「反復改善ループは効く」ことの単一論文での実証。残課題は他論文での再現性確認

§2 Self-Improving Agent

タスク
CIFAR-10 の弱いベースライン CNN（精度 65-70%）を、人手の介入なしに改善し続けて 85%+ を目指す。Claude が訓練スクリプトそのものを書き換える方式。

提案手法 — 24/365 自己改善ループ

Karpathy の autoresearch（edit → execute → measure → evaluate のループ）を発展させ、失敗履歴を構造化（NRR） + 退行時の自動ロールバックを加えた最小実装。

1 サイクルの流れ

  1. 現在の train.py で訓練 → CIFAR-10 テスト精度・損失を測定
  2. Claude が失敗分析（精度が伸びない理由）し、改善案を提案
  3. NRR（Negative Result Repository）が過去の失敗パターンを照合し、proceed / caution / avoid の判定を返す
  4. avoid なら別方向を再提案、proceed なら train.py を自動修正して再訓練
  5. 精度が best から大きく退行したら best チェックポイントから自動ロールバック
  6. 結果を cycle-NNNN.json に記録、次サイクルへ

提案カテゴリ（6 種類）

  • scheduler（学習率スケジューラ）/ regularization（dropout、GroupNorm 等）/ architecture（層追加・置換）/ optimizer（Adam の係数調整等）/ augmentation（データ拡張）/ other

NRR 統合の意義

LLM は「同じ方向の失敗」を繰り返す傾向がある（例: 学習率を上げる提案を 4 サイクル連続）。NRR がそれを検出し、avoid シグナルで別方向に強制的に切り替える。これにより探索の多様性が確保される。

結果が示すこと

  • 5 サイクルで精度が 0.6782 → 0.6957（best）まで改善（cycle 4）
  • cycle 3 と 5 は退行（より深い CNN を提案したが弱体化）— ロールバックの必要性を体現
  • 「毎サイクル改善するわけではない」「LLM の提案には偏りがある」「NRR とロールバックがないと退行が積み上がる」3 つの工学的知見を取得
  • 残課題: プラトー検出の精緻化、長時間運用での挙動確認、ドメイン拡張

§3 Reproduce Pipeline

タスク
arXiv URL を 1 つ与えるだけで、論文を解析し、コードを取得・生成し、環境を構築し、実行し、論文の数値と一致するか検証するまでを自動化する。「人がやれば 1 時間以上かかる作業」を end-to-end で実行する。

提案手法 — 5 ステージのパイプライン

研究自動化の最大ボトルネックは 環境構築（成功率 < 7% / EnvBench） と 実行検証（成功率 0% / PRBench end-to-end）。これを 5 ステージに分解し、各ステージで何が起きているかを観測可能にする。

5 ステージの設計判断

  1. Paper Fetching（〜15秒） — arXiv API で PDF 取得 → Claude が方法論・claims・hyperparameters を抽出。なぜ Claude か: 構造化抽出が LLM 得意領域、ヒューリスティックパーサーより堅牢
  2. Code Finding（〜90秒） — 公式 GitHub があればクローン、なければ Claude が論文記述から実装を生成（DeepCode/AIDE と類似のフォールバック）。なぜ生成 fallback: 論文の 60-70% は公開コード無し or 部分的
  3. Environment Building（〜0.3秒、Stochastic Interpolants では venv 即構築） — Python venv + 依存解決。なぜ Docker 不採用: arXiv 2601.12811 が「Docker でも再現性は保証されない」と示しているため、まず venv で進める
  4. Experiment Execution（〜56秒） — 訓練スクリプトを実行、GPU/CPU 自動判定、stdout/stderr を構造化キャプチャ
  5. Result Verification（<1秒） — Claude が論文の Table/Figure から数値を抽出 → 実行結果の metrics と ±5% 許容で比較。一致しなければ理由を記録

Code Finding が長い理由

5 ステージ中、Code Finding が 56%（90秒 / 161秒）を占める。これは Claude が論文を読んで一からコードを生成するため。公式コードが存在する場合は数秒で済むが、Stochastic Interpolants では生成 fallback を選択。

結果が示すこと

  • 5 ステージ全て成功、合計 161 秒で完走（人手なら数時間）
  • 再現スコア 0.5（3 claims のうち FID 数値が untested＝実行出力に該当 metric が無かった）
  • 環境構築・実行は通ったが、数値完全一致は未達。これは EnvBench/PRBench で示された業界全体のギャップと一致
  • 残課題: Stage 2 のコード生成精度（PaperBench iter で +51.3pp 改善実証済み）、Stage 5 の数値抽出強化、GPU 環境対応

§4 Literature Scout

タスク
「自分の研究テーマに関連する論文を探したい」という瞬間に、自然言語クエリ + 進行中の研究コンテキストを渡すと、関連度ランクされた論文リストを 30 秒程度で返す。

提案手法 — pull 型の論文発見

既存の Intel システム（5 collectors / 日次 210+件）は push 型（自動収集）。Scout は pull 型（オンデマンド検索）。両者を使い分ける設計。

動作の流れ

  1. クエリ + 任意の --context（進行中研究の説明）を入力
  2. arXiv API と Semantic Scholar API を並列呼出（別ソースで取りこぼし防止）
  3. 各論文の title + abstract を Claude に渡す
  4. Claude が --context に対する関連度（0.0 〜 1.0）+ 判断理由を出力
  5. 関連度でソート、上位 N 件を CLI 出力

なぜ並列検索か

arXiv は preprint 中心、Semantic Scholar は引用ネットワーク中心。同じクエリで取れる論文集合が大きく異なる。両方叩いて重複を除去することで取りこぼしを最小化。

Claude による関連度判定

title + abstract だけでは「関連あり」と「真に役立つ」の区別が難しい。--context（例: 「ML 実験を自律実行して精度改善するシステム」）を Claude に渡すことで、「この研究文脈で読む価値があるか」を判定する。判断理由も同時に出力されるので、なぜ関連と判定したかが透明。

結果が示すこと

  • 30〜60秒で関連論文リストが返る（API 呼び出し + Claude 分析）
  • Intel（push）と Scout（pull）の使い分けで「毎日勝手に入ってくる」+「気になった瞬間に深掘れる」が成立
  • 残課題: relevance 判定の精度評価、autoresearch の program.md 自動更新との連携
注意: 本ショーケースのライブ取得は、arXiv 429（rate limit）と Semantic Scholar 0 件返却で当面動かない。代表的な擬似サンプル出力を 結果ページ §4 に表示している。

§5 4 コンポーネントの関係

個別運用 vs 統合運用

4 つはそれぞれ単独で価値を生む CLI ツール。一方、組み合わせると「論文発見 → 再現 → スコア評価 → 反復改善」の end-to-end ループになる。

シナリオ使うコンポーネント
新トピックの論文を探したいLiterature Scout
気になる論文を再現したいReproduce Pipeline
再現コードを SOTA レベルに引き上げたいPaperBench Iter
新しいモデル/ハイパラを自動探索したいSelf-Improving Agent
上記を全部つなげて止まらない研究エージェントにしたい4 つ全てを統合（=「24/365 研究エージェント」の最終形）

残された大きな課題

  • 環境構築の堅牢化（業界全体で 7% 未満、ここは未踏領域）
  • 数値結果の検証強化（PRBench end-to-end は 0%）
  • 暗黙知の補完（論文に書かれない関係的・身体的・集団的知識）
  • 長時間運用（24/365 ループの安定性、メモリ管理、コスト管理）

これらは AutoRes グループとして次の 12-18 ヶ月で取り組むべき領域。各コンポーネントは個別に磨きつつ、統合運用への橋渡しを進めていく。
